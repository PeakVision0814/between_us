# 架构说明

这份文档定义了当前产品方向下，必须保持稳定的页面结构、内容边界和共享规则。

## 产品定位

当前产品不再以“片刻记录”为唯一核心，而是转向一款：

- 轻实用
- 低压力
- 双人共享
- 兼具情感温度与生活组织能力

的情侣 App。

一句话定义：
它是一个让两个人一起看时间、存计划、写随记、管理共同空间的生活型情侣 App。

## 主界面结构

一级导航统一为 4 个：

- `首页`
- `日历`
- `计划笔记`
- `我们`

这 4 个界面承担的是长期稳定的信息架构，不应轻易变动。

## 产品体验原则

- 首页常规但有温度，不故作复杂。
- 整个 App 应该让人觉得轻松，而不是被催着使用。
- 产品可以承载轻日记式内容，但不应该让人感觉“必须天天写”。
- 日历是时间入口，计划笔记是未定事项和随记入口，两者边界必须清晰。
- 我们页同时承载个人偏好与双人设置，但必须分组明确。
- 默认语言是简体中文，英文是可选项。
- 必须支持 `system / light / dark` 三种主题模式。
- 涉及健康与隐私的数据，默认谨慎，明确授权后再共享。

## 页面边界总规则

- 有明确日期的内容，归 `日历`
- 没有明确日期的计划，归 `计划笔记 > 计划`
- 自由的日常记录与随手想法，归 `计划笔记 > 随记`
- 个人设置与双人空间设置，归 `我们`
- 首页只展示状态和入口，不承担所有详情内容

## 内容归属规则

不同类型内容的协作方式不同：

- `日历事件`：双人共享内容，双方都可以查看与共同维护
- `计划`：双人共享内容，双方都可以查看和推进状态
- `随记`：双人可见，但默认只允许作者编辑和删除自己的内容
- `经期记录`：属于记录者本人，只有在明确授权共享后，另一方才能查看
- `个人偏好`：只属于个人，例如语言、主题、时区
- `双人空间设置`：属于双方共享空间，例如邀请状态、空间名称、共享规则

## 四个主界面的详细信息架构草图

### 1. 首页

#### 页面目标

- 让用户一打开就知道“今天需要先看什么”
- 让产品看起来像情侣空间，而不是工具目录
- 作为 4 个主界面的聚合入口，但不抢走其他页面职责

#### 首屏模块顺序

1. `我们概览`
   - 两人头像 / 昵称
   - 在一起天数或空间状态
   - 一句轻量状态描述
2. `下一个重要日期`
   - 最近的纪念日 / 约会 / 已确定事项
   - 倒计时或即将发生提示
3. `最近动态预览`
   - 最近一条随记，或最近一次共享更新
4. `最近一个计划提醒`
   - 当前最值得推进的一条计划
5. `快捷入口`
   - 去日历
   - 新建计划
   - 写随记
   - 去我们

#### 页面不该承载的内容

- 长列表
- 复杂筛选
- 大量历史数据
- 解释产品为什么这样设计的说明文字

#### 未来可扩展内容

- 首页卡片排序个性化
- 重要事项置顶
- 今日天气 / 节假日氛围类轻增强

---

### 2. 日历

#### 页面目标

- 成为所有“和日期明确绑定”的内容容器
- 让用户通过一个时间视图看清未来与纪念

#### 首屏模块顺序

1. `月视图日历`
   - 当前月份
   - 带标记的日期格
2. `选中日期详情`
   - 当日纪念日
   - 当日约会 / 提醒 / 计划后的事项
3. `近期事项列表`
   - 接下来 7 天或最近几件事
4. `新建日历项入口`
   - 纪念日
   - 约会安排
   - 提醒

#### 核心规则

- 只有已确定日期或时间的内容进入日历
- 纪念日、约会、提醒、定好的计划都属于日历
- 附着在某个日期上的备注，仍然归日历，不复制到计划笔记

#### 经期记录在日历中的位置

经期记录未来进入日历，但属于独立敏感分组：

- 不和纪念日混成同一种事件
- 默认不共享
- 只有记录者明确授权后，另一方才能查看
- 首版只做手动记录，不做复杂预测

#### 未来可扩展内容

- 周视图 / 议程视图
- 事件筛选
- 经期趋势与预测
- 与计划笔记更深联动

---

### 3. 计划笔记

#### 页面目标

- 承接所有还没定下日期的计划
- 承接可以随手写下来的共享随记
- 避免让用户把任何东西都塞进日历

#### 内部分区

页面内部固定拆成两个区：

1. `计划`
2. `随记`

#### `计划` 区职责

放这些内容：

- 还没有明确日期的约会想法
- 想去的地方
- 待讨论事项
- 还没决定什么时候执行的生活计划

建议模块：

- 计划列表
- 计划状态（例如：想法中 / 讨论中 / 已安排 / 已完成）
- “加入日历”动作

#### `随记` 区职责

放这些内容：

- 今天的想法
- 日常小记录
- 随手写下的文字
- 轻量共享日记

建议模块：

- 快速输入入口
- 随记时间流
- 作者标识

#### 页面核心规则

- `计划` 不是日历
- `随记` 可以像轻量日记，但不制造打卡压力
- 已经确定日期的计划应当能进入日历
- 随记不按“必须每天写”设计

#### 未来可扩展内容

- 标签
- 置顶
- 搜索
- 图片附件
- 计划转日历后的状态跟踪

### 4. 我们

#### 页面目标

- 统一承接个人偏好与双人空间设置
- 强调“这是我们共同使用的空间”，而不是普通的“我的”页面

#### 页面分区

固定拆成两个一级分组：

1. `我的偏好`
2. `我们的空间`

必要时增加：

3. `隐私与共享`

#### `我的偏好` 应包含

- 语言
- 主题
- 时区
- 通知预览

#### `我们的空间` 应包含

- 空间名称 / 关系信息
- 邀请状态
- 共享规则说明
- 共同纪念日基础信息

#### `隐私与共享` 应包含

- 敏感数据共享说明
- 经期记录共享开关或状态
- 导出与解绑入口说明

#### 页面核心规则

- 个人设置和双人设置必须分组，不能混在一长页里
- 我们页不能只做“设置堆砌页”，需要有空间归属感
- 涉及健康和敏感共享的配置必须有清晰解释

#### 未来可扩展内容

- 共享头像 / 空间封面
- 自定义纪念日规则
- 共享通知偏好
- 数据导出历史

## 登录、身份与 Onboarding

### 登录基础

- 用户必须先完成登录，才能进入完整 App 界面
- 未登录状态只展示登录流程，不进入 App Shell
- 当前阶段登录方式为邮箱验证码登录（Email OTP）
- 手机号验证码登录（SMS OTP）必须在私测前完成
- 当前阶段不做微信登录
- 同一账号只允许一台设备保持有效登录；新设备登录成功后，旧设备会话失效
- 同一账号换设备后，仍应看到同一份云端数据和同一用户身份

### 验证码服务说明

邮箱验证码和短信验证码均由 Supabase Auth 内置提供，不需要自己写后端发送逻辑。

**邮箱验证码（Email OTP）：**
- 开发阶段：Supabase 自带测试邮件服务，可发送到开发者自己的邮箱
- 上线阶段：需配置正规 SMTP 服务（如 Resend、SendGrid、阿里云企业邮箱）
- 配置位置：Supabase Dashboard → Project Settings → Auth → SMTP Settings

**短信验证码（SMS OTP）：**
- 需要接入第三方短信服务商：Twilio（国际）、阿里云短信、腾讯云短信等
- 配置位置：Supabase Dashboard → Project Settings → Auth → SMS Provider
- 填入服务商的 API Key 即可，Supabase 负责发送

### 本地数据与云端数据

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 登录 session（token） | 本地 | Supabase SDK 自动缓存在设备上，App 重启后恢复登录状态 |
| profiles、couple_spaces、日历、计划、随记 | 云端 | Supabase PostgreSQL，所有业务数据 |
| App 偏好（语言、主题） | 云端 + 本地缓存 | profiles 表存一份，AppController 内存持有一份 |

退出登录时需清理：本地 session token + AppController 内存状态。云端数据不动。

### 账号与资料基础

- `auth.users.id` 是用户的真实账号标识
- `profiles.display_name` 是用户在产品内展示给伴侣和自己看的身份标识
- 登录成功后，系统先初始化 `profiles`、偏好与账号状态
- 登录成功后立即引导补全个人资料，但这一步不等于创建单人空间
- `couple_space` 仅在用户首次写入内容或进入配对流程时再创建
- 首次登录时，如果 `display_name` 是默认值（"新的用户"），进入昵称设置引导
- 昵称设置后可在"我们"页随时修改
- 头像暂用首字母头像（取昵称第一个字），不做图片上传

### 登录与注册分离

登录和注册是两个独立流程，不允许合并：

- **登录**：仅允许已注册用户使用，输入未注册邮箱时提示”该邮箱尚未注册”并引导去注册页
- **注册**：新用户通过注册页创建账号，同样使用邮箱验证码确认身份

Supabase 配置要求：关闭 “Enable automatic account creation on sign in”，确保 signInWithOtp 不会对不存在的邮箱自动创建账号。

分离的原因：
- 用户必须明确知道自己在做什么（登录 vs 注册）
- 防止输错邮箱意外创建空账号产生脏数据
- 注册时可引导用户了解产品定位
- 为后续手机号登录/注册预留清晰的入口结构

### 登录门禁流程

```
App 启动 → 初始化 Supabase → 检查 session
  → 有 session → 直接继续初始化用户数据
  → 无 session → 进入登录页
    → 用户输入已注册邮箱 → 发送验证码 → 输入 6 位验证码 → 验证成功
    → 用户输入未注册邮箱 → 提示”该邮箱尚未注册” → 引导跳转注册页
  → 登录/注册成功后 → 检查/创建 profile → 检查 display_name
    → 默认值 → 昵称设置引导页 → 设置后进入主页
    → 已有昵称 → 直接进入主页
```

### 注册流程

```
注册页 → 用户输入邮箱 → 调用 auth.signUp → 发送验证码
  → 输入 6 位验证码 → 验证成功
  → handle_new_user() 触发器自动创建 profiles 行
  → 进入昵称设置引导页 → 设置后进入主页
```

注意：`handle_new_user()` 触发器会在 `auth.users` 插入时自动创建 `profiles` 行，但触发器执行有微小延迟。前端在注册/登录成功后查询 profiles 时，如果查不到应等待重试，而非直接报错。

### 登录页要求

- 出现在 App Shell 之前
- 默认只提供邮箱验证码登录
- 首版主流程是 App 内输入 6 位验证码
- 不采用邮件链接跳转作为主流程
- 明确提示”登录后才能进入完整界面”
- 底部提供”没有账号？去注册”跳转链接
- 未注册邮箱输入后显示友好提示并引导去注册
- 保留未来增加手机号验证码登录入口的位置

### 注册页要求

- 出现在 App Shell 之前，与登录页平级
- 标题：中文”创建账号” / 英文”Create Account”
- 简短说明产品定位（情侣共享空间）
- 邮箱输入框 + 发送验证码按钮
- 验证码输入步骤
- 底部提供”已有账号？去登录”跳转链接
- UI 风格与登录页保持一致

### 昵称引导页要求

- 全屏页面，出现在首次登录/注册后的主界面之前
- 标题：”给自己取个名字吧”
- 输入框 placeholder：”对方会看到这个名字”
- 可跳过，但跳过后仍可在”我们”页修改

## 单人模式 vs 双人模式

### 判定方式

通过 `couple_memberships` 活跃成员数判断：
- 1 人 = 单人模式（`couple_spaces.status = 'pending_partner'`）
- 2 人 = 双人模式（`couple_spaces.status = 'active'`）

### 功能分界原则

**登录是使用前提，单人是登录后的使用模式。** 配对只是让内容变成共享的，并增加作者标识。

| 功能 | 单人 | 双人 | 说明 |
|------|------|------|------|
| 写随记 | ✅ | ✅ 显示作者 | 需先登录 |
| 新建计划 | ✅ | ✅ 显示作者 | 需先登录 |
| 日历事件 | ✅ | ✅ 显示作者 | 需先登录 |
| 首页 | ✅ 显示邀请提示 | ✅ 显示双方昵称 | |
| 生成/输入邀请码 | ✅ | ❌ 已配对 | |
| 查看对方信息 | ❌ | ✅ | |
| 经期记录 | ❌ | ❌ | 阶段 8 |

### 设计意图

已登录但未配对用户可以完整体验 App 的内容功能，了解产品价值后愿意拉上对象使用。配对不是进入 App 的前提，但登录是。

## Couple Space 生命周期

共享数据模型基于一个私密的 `couple space`，其中只允许存在 2 个活跃成员。

生命周期规则：

1. 一位已登录用户创建 `couple space`
2. 该用户邀请另一位用户加入
3. 第二位用户加入后，空间成为有效双人空间
4. 达到 2 人后不允许再加入更多成员
5. 任一成员可发起解绑，但真正删除必须明确确认

## 应用状态管理与初始化流程

### 设计原则

AppController 是整个应用的唯一状态中心。所有用户身份、空间归属、成员关系数据，统一由 AppController 持有和加载。下游页面只读取 AppController 的状态，不独立查询 Supabase。

这样做的原因：
- 避免多个页面各自初始化空间数据导致的竞态条件
- 避免重复查询造成的性能浪费和状态不一致
- 确保用户进入主界面时，所有基础数据已就绪

### 统一 Bootstrap 流程

```
App 启动
  → 初始化 Supabase SDK
  → 检查 session
    → 无 session → 登录门禁
    → 有 session → 进入数据初始化
  → 数据初始化（loadPreferences）
    → 加载 profile（display_name、偏好设置）
    → 查询 couple_memberships 获取 currentSpaceId
    → 如果没有活跃空间 → 调用 create_couple_space 创建
    → 如果有空间 → 查询同空间另一成员的 display_name
    → 设置 appReady = true
  → appReady 后进入 App Shell
```

### AppController 持有的核心状态

| 字段 | 类型 | 含义 |
|------|------|------|
| selfProfileId | String? | 当前登录用户的 UUID |
| displayName | String? | 当前用户的昵称 |
| currentSpaceId | String? | 当前用户所属的 couple_space ID |
| memberCount | int | 当前空间的活跃成员数（0/1/2） |
| partnerDisplayName | String? | 对方的昵称，仅双人时有值 |
| appReady | bool | 所有基础数据加载完毕，可安全使用功能 |

### 数据所有权规则

| 数据 | 所有者 | 下游页面如何获取 |
|------|--------|-----------------|
| 用户身份（profile） | AppController | AppScope.of(context).displayName |
| 空间 ID | AppController | AppScope.of(context).currentSpaceId |
| 成员数 / 配对状态 | AppController | AppScope.of(context).memberCount |
| 对方昵称 | AppController | AppScope.of(context).partnerDisplayName |
| 日历事件列表 | 各页面自行查询 | 使用 AppController.currentSpaceId 作为过滤条件 |
| 计划/随记列表 | 各页面自行查询 | 使用 AppController.currentSpaceId 作为过滤条件 |

### 禁止模式

- 禁止各页面自行创建 CoupleSpaceGuard 实例或独立查询空间归属
- 禁止各页面自行判断"用户是否就绪"，统一依赖 appReady
- 禁止在 AppController 之外持有 spaceId 的缓存副本
- 禁止写入数据时自行 ensure 空间，应直接使用 AppController.currentSpaceId

### 写入数据时的 spaceId 来源

所有需要 `couple_space_id` 的写入操作（创建日历事件、计划、随记、邀请码），统一从 `AppScope.read(context).currentSpaceId` 获取。如果为 null（理论上不应发生，因为 appReady 保证了），显示友好错误提示并引导用户检查网络，而非"请重启 App"。

## 敏感数据规则

经期记录属于健康相关敏感数据，必须遵循：

- 默认不共享
- 明确授权后才共享
- 只有记录者本人可以编辑
- 另一方仅在被授权时查看
- 首版不提供医疗建议式文案

## 非目标

- 把 App 做成情侣社交平台
- 把计划笔记做成复杂项目管理工具
- 在首版里就做复杂经期预测
- 在页面未稳定前扩张太多功能模块
