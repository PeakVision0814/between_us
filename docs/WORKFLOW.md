# 开发工作流

这份工作流基于当前新的 4 页面产品结构制定：

- 首页
- 日历
- 计划笔记
- 我们

目标是确保团队按“先打地基、再做核心、后做增强”的节奏推进，而不是想到什么做什么。

## 1. 先判断任务属于哪一层

每个任务在开始前，都先归类：

- `地基任务`
  - 视觉系统
  - 语言切换
  - 主题模式
  - 页面容器
  - 导航结构

- `核心产品任务`
  - 首页
  - 日历
  - 计划笔记
  - 我们

- `共享基础任务`
  - 认证
  - 邀请
  - 共享存储
  - `RLS`
  - 隐私控制

- `后期增强任务`
  - 经期增强
  - 提醒扩展
  - 照片 / 愿望清单 / 旅行等模块

规则：

- 一次只推进一个明确层级，不要跨层乱做
- 地基没稳定前，不要急着做复杂增强

## 2. 先改文档，再改代码

如果任务改变了产品结构、术语、页面职责或数据边界，必须同步改文档：

- `docs/ROADMAP.md`
- `docs/ARCHITECTURE.md`
- `docs/DATABASE.md`

禁止出现“代码已经变了，但规划还停留在旧版本”的情况。

## 3. 开发顺序必须按页面优先级推进

当前推荐顺序：

1. 视觉与壳层地基
2. 首页
3. 我们
4. 日历
5. 计划笔记
6. 共享后端
7. 敏感数据与增强功能

原因：

- 首页决定第一印象和产品温度
- 我们页承接语言、主题、共享规则，是地基页
- 日历是最清晰的功能入口，先简单做稳
- 计划笔记边界最复杂，必须在前面三者稳定后再做

## 4. 采用纵向切片，不要横向撒网

每个功能按下面顺序推进：

1. 页面职责定义
2. 文案与状态设计
3. 本地静态原型
4. 本地交互
5. 测试
6. 后端接入
7. 稳定性补强

例子：

- `日历 v1`
  - 先定义什么算日历内容
  - 再出月视图和详情区
  - 再补本地创建与展示
  - 最后才考虑共享同步

- `计划笔记 v1`
  - 先定义 `计划` 和 `随记` 的边界
  - 再做两区布局
  - 再做“加入日历”联动

## 5. 页面开发必须覆盖的状态

每个主界面在进入下一阶段前，至少覆盖：

- 默认有内容状态
- 空状态
- 首次使用状态
- 本地占位状态
- 浅色主题
- 深色主题
- 中文文案
- 英文文案

这条要求对 `首页 / 日历 / 计划笔记 / 我们` 都成立。

## 6. 文案规则

产品文案必须符合以下约束：

- 默认简体中文
- 英文作为可选语言存在
- 不写 PM 面向团队的说明文
- 不制造任务感、打卡感、日记压力
- 优先使用生活化、情侣化、自然的语言

特别是：

- 首页不能像功能目录
- 计划笔记不能像办公软件
- 我们页不能像杂乱的设置堆

## 7. 敏感数据开发规则

经期记录这类敏感数据必须遵守：

- 不在地基阶段直接实现复杂版本
- 首版只做手动记录
- 默认不共享
- 必须有明确授权
- 页面文案避免监控感、医疗建议感

如果某个任务涉及这类数据，必须同时检查：

- 产品边界
- 数据权限
- UI 提示是否足够清晰

## 8. 测试工作流

一个任务在视为完成前，应尽量完成以下检查：

- 运行 `flutter analyze`
- 运行 `flutter test`
- 更新或新增 widget tests
- 如果涉及时间、权限、状态流转，则补 pure logic tests

重点测试方向：

- 默认中文是否生效
- 主题切换是否生效
- 首页主路径是否顺畅
- 日历和计划笔记边界是否符合预期
- 敏感数据共享是否不会误暴露

## 9. 评审检查清单

每次评审都要回答：

- 这次改动是在打地基，还是在叠增强？
- 它是否符合四主界面的当前边界？
- 它是否让产品更像情侣生活空间，而不是普通工具？
- 它是否保留了低压力感？
- 它是否引入了新的隐私风险？
- 它是否把后期功能提前塞进核心路径？

## 10. 可合并标准

一个功能切片可以合并，当且仅当：

- 对应文档已同步
- 页面职责没有跑偏
- 状态覆盖足够完整
- 文案语气符合产品气质
- 测试通过，或未通过原因已明确说明
- 这次改动在推进一个清晰切片，而不是无边界扩 scope
