# 开发工作流

这份工作流定义了：从产品决策到代码实现再到验证完成，工作在这个仓库里应该如何推进。

## 1. 先判断是否符合产品方向

写代码前，先把任务归类到以下三类之一：

- `Core MVP`：直接改进 `Home`、`Moments`、`Dates` 或 `Settings`
- `Shared foundation`：认证、邀请、共享存储、`RLS`、隐私或可靠性等 `MVP` 基础能力
- `Backlog`：不属于当前约定核心闭环的任何内容

规则：

- 如果任务属于 backlog，没有新的产品决策前，不要把它推进到活跃开发。
- 如果任务会改变产品行为、术语或页面职责，必须在同一分支先同步文档。

## 2. 保持产品规则稳定

只要任务改变了产品行为，就要在改代码前或改代码时同步更新对应文档：

- `docs/ROADMAP.md`：阶段目标或里程碑发生变化时
- `docs/ARCHITECTURE.md`：产品规则、命名、信息架构或页面职责发生变化时
- `docs/DATABASE.md`：数据结构、归属、权限或设置持久化规则变化时

禁止通过代码“偷偷引入”与文档冲突的新规则。

## 3. 采用纵向切片实现

一个有意义的产品改动，应该按下面的顺序推进：

1. 产品规则
2. 体验文案与状态设计
3. 本地 UI 原型
4. 本地交互与测试
5. 共享后端接入
6. 稳定性与润色

例子：

- 一个新的首页卡片，应该先定义产品意图与文案，再做本地 UI，再补空状态，最后再接真实共享数据。
- 一个新的设置项，应该先定义归属和偏好规则，再做本地状态，最后再决定是否进入后端持久化。

## 4. 每个任务都围绕一个用户结果

好的任务形状：

- 让首页更像状态页，而不是欢迎页
- 增加中文优先、英文可选的语言支持
- 增加 `system / light / dark` 主题模式
- 把当前 `Timeline` 实现收敛到 `Moments（片刻）` 产品面

不好的任务形状：

- 顺手优化很多页面
- 整体美化一下 App
- 一次性加多个互不相关的模块

## 5. 用户可见改动必须覆盖的状态

任何面向用户的 `MVP` 改动，在完成前都必须考虑这些状态：

- 有内容状态
- 空状态
- 首次使用或接近 onboarding 的状态
- 适用时的错误状态
- 浅色主题
- 深色主题
- 中文文案
- 英文文案

这条要求特别适用于 `Home`、`Moments`、`Dates`、`Settings`。

## 6. 文案与语气规则

所有产品文案都必须遵守这些规则：

- 默认使用简体中文
- 英文作为用户可选设置保留
- 不在 UI 里写面向产品经理或开发者的解释性文案
- 避免日记感、打卡感、额度感或让人内疚的表达
- 优先使用平静、直接、自然的人类语言，而不是功能宣传语

如果某段解释只对开发者有用，就放到文档里，不要放到页面里。

## 7. 设置相关的开发流程

新增设置项时，必须先明确它属于哪一类：

- 仅本地原型使用
- 登录后属于个人并可同步
- 涉及隐私，因此必须在后端规则明确后才能上线

当前 `MVP` 的最小设置集是：

- 语言
- 主题
- 时区
- 锁屏预览行为

## 8. 测试工作流

一个任务在视为完成前，应尽量完成以下检查：

- 运行 `flutter analyze`
- 运行 `flutter test`
- 为导航或用户可见行为补充或更新 widget tests
- 如果任务修改了倒计时、权限或数据规则，则补 pure logic tests

如果当前环境里无法运行这些命令，必须在最终说明里明确写出。

## 9. 评审检查清单

每次评审都应该回答这些问题：

- 这次改动是否强化了当前约定的 `MVP` 闭环？
- 它是否保持了产品“低压力、非日记式”的气质？
- 它是否仍然符合“中文优先、英文可选”的方向？
- 它在 `system`、浅色、深色三种主题下是否都正常？
- 它是否尊重了内容归属、隐私和 `couple space` 边界？
- 它是否把 backlog 功能偷偷带进了 `MVP` 主路径？

## 10. `MVP` 改动的可合并标准

一个改动可以合并，当且仅当：

- 文档和实现保持一致
- 用户可见文案符合产品语气
- 当前阶段要求的状态已经覆盖到位
- 测试和分析通过，或者未通过原因已被明确说明
- 这次改动推动的是一个明确产品切片，而不是无边界地扩 scope
