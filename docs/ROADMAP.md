# 路线图

这份路线图围绕一个新的核心产品问题展开：
两个人会不会愿意长期使用一个轻实用、低压力的情侣共享空间，
在里面查看首页状态、管理日历、沉淀计划与随记，并维护“我们”的共同设置？

## 当前产品方向

当前版本的一级主界面统一为 4 个：

- `首页`
- `日历`
- `计划笔记`
- `我们`

这套结构对应的是一款“轻实用型情侣 App”，而不是纯粹的情绪记录工具。
它的目标不是要求用户每天打卡，而是让两个人在安排生活、记住重要日子、
记录日常想法时，愿意自然地回到这个共享空间。

## 指导原则

- 首页保持常规、稳定、好懂，但必须有情侣感和温度。
- 日历先做简单版本，再逐步扩展复杂能力。
- 计划笔记必须先划清边界，再开发功能。
- 我们页同时承接个人设置和双人共享设置。
- 默认语言为简体中文，英文为可选项。
- 主题模式属于基础能力，必须支持 `system / light / dark`。
- 产品整体保持低压力、非任务感、非强打卡感。
- 敏感数据必须显式授权共享，尤其是经期记录。
- 所有阶段必须先打地基，再做增强，不同时推进过多概念。

## 页面边界总规则

- `日历`：只放已经有明确日期或时间的内容。
- `计划笔记`：只放还没有明确日期的计划，以及不要求按时间组织的随记。
- 一旦某个计划确定了日期，应允许它进入 `日历`。
- 附着在具体日期上的备注，主归属仍然是 `日历`，不要复制成独立笔记。
- `我们`：个人偏好与双人空间设置放在同一页，但要分组清晰。

## 开发总顺序

推荐按以下顺序推进，不要跳步：

1. 视觉与壳层地基
2. 首页与我们页
3. 日历基础版
4. 计划笔记基础版
5. 共享数据规则与隐私边界
6. 登录与账号基础
7. 共享同步 Alpha
8. 用户资料与双人身份展示
9. 敏感数据能力与生活增强
10. 稳定性与私测 Beta

## 阶段 0：项目基础

目标：准备仓库和本地开发环境。

范围：

- 创建 Flutter 仓库和 Android 优先的 App 基础壳。
- 补齐许可证、路线图和开发说明文档。
- 验证本地 Flutter 工具链。

完成标准：

- 仓库能在本地 Android 环境中构建。
- 核心文档存在且可读。
- 贡献者机器上可以成功运行 `flutter doctor`。

## 阶段 1：视觉与壳层地基

目标：先把 4 个主界面的视觉和结构地基搭起来。

范围：

- 确认 4 个一级界面：`首页 / 日历 / 计划笔记 / 我们`
- 定义 UI 视觉方向、配色系统、字体层级、卡片样式、按钮样式
- 支持简体中文默认和英文切换能力
- 支持 `system / light / dark` 三种主题
- 建立统一的导航壳、页面容器和空状态规范
- 保留所有页面为本地静态数据

完成标准：

- App 首次打开默认是中文界面
- 4 个一级入口都已出现在导航中
- 视觉风格已经统一，不再是“想到什么写什么”
- 浅色、深色、跟随系统三种主题都能正常切换

## 阶段 2：首页与我们页基础版

目标：先把产品最有感觉的入口和最基础的设置承载页做对。

范围：

- 首页做成稳定的状态页，而不是产品说明页
- 首页展示情侣概览、最近动态预览、下一个重要日期、最近一个计划提醒
- 首页提供清晰快捷入口，但不堆长列表
- 我们页分成“我的偏好”和“我们的空间”两大区
- 我们页先本地承接语言、主题、时区、通知预览、邀请状态占位

完成标准：

- 用户第一次打开时，5 到 10 秒内能理解首页在展示什么
- 首页不再出现大量 PM 解释型文案
- 我们页能清楚区分个人设置与双人设置
- 首页和我们页的中文文案已经符合真实用户语气

## 阶段 3：日历基础版

目标：先把“有明确日期的事情”统一收进一个简单可用的日历里。

范围：

- 月视图日历基础框架
- 日期详情区：展示某天的纪念日、约会、提醒、已排定事项
- 本地支持创建最简单的日历项
- 首版只做简单的日期与事件展示，不追求复杂交互
- 为经期记录预留产品位置，但不一开始就做复杂能力

完成标准：

- 用户可以一眼看出最近的重要日期和安排
- 某一天被点击后，可以看到该日的内容列表
- 纪念日、约会、提醒等有明确日期的事项已经能被纳入日历
- 页面结构清楚，不会和计划笔记混淆

## 阶段 4：计划笔记基础版

目标：把“还没定日期的计划”和“轻量共享随记”沉淀到一个统一页面里。

范围：

- 页面名称保留为 `计划笔记`
- 页面内部明确拆分为 `计划` 和 `随记` 两个区
- `计划`：承接还没有明确日期的想法、约会意向、待讨论事项
- `随记`：承接日常想法、临时记录、轻量共享日记
- 提供从 `计划` 进入 `日历` 的后续联动预留

完成标准：

- 用户能理解“有日期去日历，没日期来计划笔记”
- `计划` 和 `随记` 两个区不会互相混淆
- 随记能呈现轻松、随手写的感觉，而不是正式日记本
- 计划笔记不再是一个什么都往里塞的总收纳箱

## 阶段 5：共享数据规则与隐私边界

目标：在开始后端开发前，先把数据模型和隐私规则定死。

范围：

- 编写并确认 `首页 / 日历 / 计划笔记 / 我们` 的数据归属规则
- 明确日历事件、计划、随记、共享设置的数据结构方向
- 明确邀请流程和 `couple space` 生命周期
- 明确双人共享设置与个人偏好的边界
- 明确经期记录的共享规则：必须显式授权

完成标准：

- 共享数据结构在文档中已经稳定
- 页面边界与数据边界一致
- 邀请、解绑、删除、导出规则有书面说明
- 敏感数据共享规则有明确约束，不靠默认猜测

## 阶段 6：登录与账号基础

目标：把“必须先登录才能进入完整界面”做成稳定的产品前提，并建立可持续的账号体系。

背景：当前实现依赖匿名登录直接进入 App，这不符合产品要求。新的基线是：未登录用户只能看到登录流程，登录成功后才进入 App Shell。当前阶段先做邮箱验证码登录，后续必须补手机号验证码登录。

范围：

- 接入 Supabase Auth 的邮箱验证码登录（Email OTP）
- 新增登录页、验证码页、登录中与失败态
- App 启动时先检查 session；无 session 时进入登录门禁
- 登录成功后先初始化 profile、偏好与账号状态；不把这一步视为创建单人空间
- 邮箱登录首版采用 App 内输入 6 位验证码
- 支持退出登录；退出时清理本地 session token 和 AppController 内存状态，云端数据不动
- 配置 Supabase 邮件发送服务：开发阶段用 Supabase 自带测试邮件；上线前配置正规 SMTP（如 Resend、SendGrid、阿里云企业邮箱），在 Supabase Dashboard → Project Settings → Auth → SMTP Settings 填入
- 登录成功后查询 profiles 时注意 `handle_new_user()` 触发器的时序：如果 profiles 行尚未就绪，需等待或重试，不要直接报错
- 同一账号只允许一台设备保持有效登录；新设备登录成功后，旧设备会话失效，但云端数据仍属于同一账号

待补完（登录与注册分离）：

当前实现中 Supabase Auth 的 signInWithOtp 会在邮箱不存在时自动创建账号，导致登录和注册混为一体。这不符合产品预期，必须拆分为两个独立流程：

- 关闭 Supabase “Enable automatic account creation on sign in” 配置
- 登录页只允许已注册用户登录；未注册邮箱返回明确提示并引导去注册
- 新增独立注册页面，注册流程同样使用邮箱验证码
- 注册成功后进入现有的昵称设置引导流程
- 登录页和注册页之间可互相跳转

分离的原因：
- 用户需要明确知道自己在做什么（登录 vs 注册）
- 防止输错邮箱意外创建空账号
- 注册时可引导用户了解产品定位（情侣共享空间）
- 为后续手机号登录/注册预留清晰的入口结构

阶段边界：

- 本阶段只做邮箱验证码登录和注册，不做手机号验证码登录 UI
- 本阶段不采用邮件链接跳转作为主流程
- 本阶段不做微信登录
- 本阶段不改变情侣配对规则，只解决”谁能进入 App”
- 手机号验证码登录必须在阶段 10（私测）之前完成，具体放在哪个阶段由开发节奏决定

完成标准：

- 未登录用户不能进入完整主界面
- 用户可通过邮箱验证码完成登录
- 未注册邮箱在登录页输入后，显示”未注册”提示并引导去注册
- 新用户可通过注册页完成注册（邮箱验证码）
- 注册成功后进入昵称设置引导流程
- 重新打开 App 时已登录 session 可以恢复
- 用户可主动退出登录并回到登录页；退出后本地 session 和内存状态已清理
- 账号状态错误有明确提示，不静默失败
- 邮件验证码在开发环境可正常收发

## 阶段 7：共享 Alpha

目标：在正式登录前提下，把这 4 个主界面的核心数据接上真实共享能力。

范围：

- 登录后接入 `couple space` 邀请流程
- 日历事件支持双人共享同步
- 计划与随记支持双人共享同步
- 我们页中的个人设置和共享设置支持持久化
- 首页聚合真实共享数据

完成标准：

- 两个已登录账号可以进入同一个共享空间
- 日历、计划、随记在两台设备间可以正确同步
- 首页可以展示真实的共享聚合信息
- 数据访问遵守 `RLS` 和空间隔离

## 阶段 8：用户资料与双人身份展示

目标：在正式登录和共享链路稳定后，把”我是谁、对方是谁、谁写了什么”展示完整。

前置重构（数据源收敛）：

在推进本阶段的功能需求之前，必须先完成应用状态管理的收敛重构：
- 将 couple space 的初始化和数据所有权统一收归 AppController
- 移除各页面独立的 CoupleSpaceGuard 实例（UsScreen、TimelineScreen、AnniversariesScreen）
- AppController 在 bootstrap 阶段确保空间存在，暴露 appReady 状态
- 下游页面统一从 AppController 读取 currentSpaceId，不再各自查询
- 消除邀请码生成等操作中的空间初始化竞态问题

此重构不改变用户可见行为，但为后续所有需要 spaceId 的功能提供可靠基础。
详见 `docs/ARCHITECTURE.md` 中”应用状态管理与初始化流程”章节。

范围：

- 首次登录后引导设置昵称（如果 `display_name` 是默认值则进入引导页）
- `AppController` 增加用户身份状态（selfProfileId、selfDisplayName、currentSpaceId、memberCount、partnerDisplayName）
- 首页 Hero 区域从硬编码改为动态读取真实昵称
- 单人模式下首页显示邀请提示卡片，引导用户配对
- 双人模式下首页显示双方真实昵称
- 我们页显示当前用户昵称（可编辑），双人时显示对方昵称
- 邀请区域根据配对状态显示/隐藏
- 随记、计划、日历事件在双人模式下显示作者昵称
- RLS 策略：允许同一 `couple space` 的活跃成员互相看到 `profiles`

功能分界原则：

- 登录是进入 App 的前提，不登录不能使用内容功能
- 登录后，单人模式仍可使用随记、计划、日历
- 配对后内容变成共享，并增加作者标识
- 邀请/配对功能在单人时可见，双人后隐藏
- 查看对方信息仅双人可用

完成标准：

- 用户首次登录后会看到昵称设置引导
- 首页显示用户真实昵称，不是硬编码占位名
- 单人用户在已登录状态下可以正常使用随记、计划、日历
- 单人首页有明确的邀请入口，引导配对
- 配对后首页显示双方昵称
- 配对后随记/计划/日历事件显示作者
- 我们页能区分单人/双人状态，展示对应信息

## 阶段 9：敏感数据能力与生活增强

目标：在共享能力稳定后，再逐步增加更敏感、更贴近日常生活的功能。

范围：

- 经期记录首版
- 仅支持手动记录，不做复杂预测
- 男生可见的前提是女生明确开启共享
- 在日历中展示经期相关信息，但和纪念日、约会保持视觉区分
- 可逐步增加计划提醒、约会提醒等增强能力

完成标准：

- 经期记录可以手动添加和查看
- 共享权限逻辑明确且可控
- 文案和交互不会让用户产生被监控感
- 敏感数据不会因为默认设置而意外暴露

## 阶段 10：稳定性与私测 Beta

目标：在扩展更多模块前，先验证整体产品是否真的被情侣自然使用。

前提：手机号验证码登录（SMS OTP）必须在本阶段开始前完成。国内私测用户习惯手机号登录，仅靠邮箱验证码不够。

范围：

- 完善关键逻辑测试、共享权限测试、时间逻辑测试
- 改善离线行为、错误提示和重试机制
- 复核中英文文案质量和三种主题体验
- 小范围私测，收集真实情侣反馈
- 重点验证首页、日历、计划笔记、我们这 4 个页面的使用频率和边界清晰度

私测重点观察：

- 首页是不是足够顺手
- 日历和计划笔记会不会混
- 我们页是否真的能承接个人与双人设置
- 经期记录共享是否让用户感到安心
- 用户会不会觉得它太工具，还是刚好实用

完成标准：

- 关键场景的失败状态可见且可恢复
- 测试覆盖关键逻辑
- 私测反馈能产出明确的保留、调整、删除决策
- 团队可以基于证据决定下一批扩展模块
